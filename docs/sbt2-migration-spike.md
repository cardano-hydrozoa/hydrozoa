## TL;DR

A full build cycle ‚Äî **plugin resolution ‚Üí build load ‚Üí compile ‚Üí test-compile ‚Üí test run ‚Üí
Docker packaging** ‚Äî works under **sbt 2.0.1** with the existing **Nix** toolchain and a
**small, contained** set of build changes (5 files). **No hard blockers found.** The only real
unknown is a scalafix `ExplicitResultTypes` output delta. sbt 2.0.0 is only ~1 month old
(released 2026-06-14), so "GO whenever convenient" is the right posture ‚Äî the change is cheap and
fully reversible.

## Build changes required (the whole migration diff)

**`project/build.properties`** ‚Äî `sbt.version=1.11.6` ‚Üí `2.0.1`

**`project/plugins.sbt`** ‚Äî bump to sbt-2 artifacts, **drop `sbt-bloop`**:
- scalafmt `2.5.2`‚Üí`2.6.1`, native-packager `1.11.1`‚Üí`1.11.7`, jmh `0.4.7`‚Üí`0.4.8`, scalafix stays `0.14.7`
- removed `sbt-bloop 2.0.5` (see Bloop note)

**`build.sbt`** ‚Äî four categories of change:
1. **Removed sbt-0.13 syntax** ‚Äî `excludeFilter in Global` ‚Üí `Global / excludeFilter`.
2. **Scala 3.8 infix** ‚Äî `... % scalusVersion withSources ()` (infix nullary + unit) is now a
   hard error ‚Üí `(... % scalusVersion).withSources()`. (`cross CrossVersion.full` only warns.)
3. **Bare-settings-apply-to-all-subprojects** (sbt 2's biggest structural change) ‚Äî top-of-file
   bare settings now leak into *every* subproject, not just root. Moved the root-app settings
   (`enablePlugins(JavaAppPackaging, DockerPlugin)`, `mainClass`, all Docker settings) into the
   `core` project's `.settings(...)`. Removed the now-redundant per-project `addCompilerPlugin`
   in `cardanoOnchain` (the top-level bare one now covers all subprojects).
4. **native-packager 1.11.7 behavior** ‚Äî multi-stage `FROM <img> AS <stage>` carries several
   args, so `case Cmd("FROM", _)` stopped matching and `STOPSIGNAL` was silently dropped from the
   Dockerfile. Fixed to `case Cmd("FROM", _*)`. *(Coupled to the migration: 1.11.1 has no sbt-2
   artifact, so the bump is forced.)*

## Notes / follow-ups (none blocking)

- **Bloop:** added incidentally by Ilia (Oct 2025), not Peter; not referenced in CI or docs.
  Metals already runs on the **sbt BSP server** (`.bsp/sbt.json`, updated Jul 8); the
  `.metals/.reports/bloop/` show bloop *erroring* on Jul 2 before the switch. The three
  Metals-autogenerated `project/**/metals.sbt` files pin `sbt-bloop 2.0.19` (no sbt-2 artifact) ‚Äî
  parked aside for the spike. **Landing requires** Metals to keep using sbt BSP (it does) and not
  re-inject bloop; otherwise `bloopInstall` breaks the sbt-2 build.
- **scalafix `ExplicitResultTypes`:** plugin runs fine, but `--check` reports **26 diffs** on
  source that is *identical to main* with the *same scalafix 0.14.7* ‚Äî i.e. the rule is sensitive
  to the sbt-2 semanticdb/compile environment. Needs a baseline decision: apply `scalafixAll`
  (mechanical reformat) or investigate the semanticdb delta. **Biggest unknown, but bounded.**
- **`target/` layout changed** ‚Üí `target/out/jvm/scala-<ver>/<project>/‚Ä¶`. Check `.gitignore`
  (`target/` still covers it), CI cache keys, and any doc/tooling that hard-codes `target/‚Ä¶`.
- **Nix:** no launcher change needed. Minor: the `flake.nix` pre-commit hook runs
  `sbt "scalafixAll --check" scalafmtCheck` (will fail until the scalafix delta is resolved); the
  bloop comment is stale; confirm `sbtn` thin-client (nix 1.11.7) vs the sbt-2 server or switch
  scripts to `sbt`.
- **Cached tasks (sbt 2):** side-effecting tasks may need `Def.uncached`. Nothing hit in this
  spike, but worth a scan of any custom tasks before landing.
- **Docs:** CLAUDE.md `sbtn` references, `just build-werror` path, and target-path mentions.

## Recommendation

**GO ‚Äî technically feasible now, low risk, not urgent.** Nothing is blocking; the migration diff
is small and reversible. Given sbt 2 is ~1 month old, land it when convenient (or after the
scalafix delta is understood), rather than treating it as urgent. A "wait a quarter for the
ecosystem to settle" posture is also defensible but not required ‚Äî every plugin this project uses
already ships sbt-2 artifacts.
