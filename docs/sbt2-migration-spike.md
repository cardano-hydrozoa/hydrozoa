# sbt 1.11 → sbt 2 migration — feasibility spike report

**Branch:** `spike/sbt2` (off `main`)  ·  **Date:** 2026-07-09  ·  **Verdict: GO (low-risk, not urgent)**

## TL;DR

A full build cycle — **plugin resolution → build load → compile → test-compile → test run →
Docker packaging** — works under **sbt 2.0.1** with the existing **Nix** toolchain and a
**small, contained** set of build changes (5 files). **No hard blockers found.** The only real
unknown is a scalafix `ExplicitResultTypes` output delta. sbt 2.0.0 is only ~1 month old
(released 2026-06-14), so "GO whenever convenient" is the right posture — the change is cheap and
fully reversible.

## Build changes required (the whole migration diff)

**`project/build.properties`** — `sbt.version=1.11.6` → `2.0.1`

**`project/plugins.sbt`** — bump to sbt-2 artifacts, **drop `sbt-bloop`**:
- scalafmt `2.5.2`→`2.6.1`, native-packager `1.11.1`→`1.11.7`, jmh `0.4.7`→`0.4.8`, scalafix stays `0.14.7`
- removed `sbt-bloop 2.0.5` (see Bloop note)

**`build.sbt`** — four categories of change:
1. **Removed sbt-0.13 syntax** — `excludeFilter in Global` → `Global / excludeFilter`.
2. **Scala 3.8 infix** — `... % scalusVersion withSources ()` (infix nullary + unit) is now a
   hard error → `(... % scalusVersion).withSources()`. (`cross CrossVersion.full` only warns.)
3. **Bare-settings-apply-to-all-subprojects** (sbt 2's biggest structural change) — top-of-file
   bare settings now leak into *every* subproject, not just root. Moved the root-app settings
   (`enablePlugins(JavaAppPackaging, DockerPlugin)`, `mainClass`, all Docker settings) into the
   `core` project's `.settings(...)`. Removed the now-redundant per-project `addCompilerPlugin`
   in `cardanoOnchain` (the top-level bare one now covers all subprojects).
4. **native-packager 1.11.7 behavior** — multi-stage `FROM <img> AS <stage>` carries several
   args, so `case Cmd("FROM", _)` stopped matching and `STOPSIGNAL` was silently dropped from the
   Dockerfile. Fixed to `case Cmd("FROM", _*)`. *(Coupled to the migration: 1.11.1 has no sbt-2
   artifact, so the bump is forced.)*

## Notes / follow-ups (none blocking)

- **Bloop:** added incidentally by Ilia (Oct 2025), not Peter; not referenced in CI or docs.
  Metals already runs on the **sbt BSP server** (`.bsp/sbt.json`, updated Jul 8); the
  `.metals/.reports/bloop/` show bloop *erroring* on Jul 2 before the switch. The three
  Metals-autogenerated `project/**/metals.sbt` files pin `sbt-bloop 2.0.19` (no sbt-2 artifact) —
  parked aside for the spike. **Landing requires** Metals to keep using sbt BSP (it does) and not
  re-inject bloop; otherwise `bloopInstall` breaks the sbt-2 build.
- **scalafix `ExplicitResultTypes` delta — RESOLVED (understood + fixed).**
  - **Symptom:** under sbt 2, `scalafixAll --check` reports **26 diffs** on source *identical to
    main* with the *same scalafix 0.14.7*. Baseline confirmed: on `main` under sbt 1.11.6 the
    same check **passes** (0 diffs) — so the delta is genuinely sbt-2-induced, not pre-existing.
  - **Why:** `ExplicitResultTypes` on Scala 3 infers each definition's type with an in-process
    Scala-3 presentation compiler (this project even forces matching compiler artifacts via
    `.scalafix.conf`'s `fetchScala3CompilerArtifactsOnVersionMismatch` to stop it crashing on the
    scalus plugin — "key not found: posttyper"). Whether that pc can type a given definition
    depends on the classpath/SemanticDB target sbt hands scalafix. sbt 2's pipeline (new
    `target/out/...` layout, cleaner classpath) lets the pc successfully infer — and therefore
    require explicit annotations on — 26 public members that sbt 1's fragile run left un-inferred
    and silently skipped. **The suggestions are legitimate; sbt 1 was under-enforcing the rule.**
  - **Verification:** applied `scalafixAll` (adds `: TxTiming`, `: Int`, `: Partition`, `: String`,
    … to 26 public defs/vals across 14 files) → recompiles cleanly and `--check` now passes.
    Needed two scalafix passes to converge (ERT qualifies a type → OrganizeImports drops the
    now-unused import on the next pass — normal multi-rule behavior, not sbt-2-specific).
  - **Decision to make when landing:** keep the 26 explicit-type annotations (recommended — they
    are correct public-API types), or relax the rule in `.scalafix.conf` (e.g. `memberVisibility`
    / `skipSimpleDefinitions`) if the team prefers to defer.
- **`target/` layout changed** → `target/out/jvm/scala-<ver>/<project>/…`. Check `.gitignore`
  (`target/` still covers it), CI cache keys, and any doc/tooling that hard-codes `target/…`.
- **Nix — cleaned up.** No launcher change needed: the nixpkgs sbt 1.x launcher bootstraps sbt 2
  from `build.properties`. Done:
  - Pre-commit hook **fixed** — sbt 2 concatenates multiple program args into one command line,
    so `sbt "scalafixAll --check" scalafmtCheck` was parsed as a single command (`scalafmtCheck`
    became a scalafix rule → `Unknown rule 'scalafmtCheck'`). Now `sbt "; scalafixAll --check ;
    scalafmtCheck"`.
  - Stale `GraalVM … metals and bloop only` comment removed (the jdk is `openjdk25`, not GraalVM).
  - **`sbtn` caveat documented** — the nix-bundled `sbtn` is the sbt **1.x** native thin client
    and cannot drive an sbt 2 server (it loops on `unknown event: sbt/exec` and never returns).
    Use `sbt`, not `sbtn`, under sbt 2 until nixpkgs ships an sbt 2 package. *(Follow-up for
    landing: update the CLAUDE.md "use sbtn for faster execution" note accordingly.)*
- **Metals — cleaned up.** Metals runs on the sbt BSP server (`.bsp/sbt.json`), so it should not
  regenerate the `sbt-bloop`-injecting `project/**/metals.sbt` files. Those (now `**/metals.sbt`)
  are **gitignored** so they can't be committed or break the sbt 2 build. If a dev's Metals is set
  to the *Bloop* build server it will re-create them with the sbt-2-incompatible `sbt-bloop
  2.0.19` — keep Metals on the sbt build server.
- **Cached tasks (sbt 2):** side-effecting tasks may need `Def.uncached`. Nothing hit in this
  spike, but worth a scan of any custom tasks before landing.
- **Docs:** CLAUDE.md `sbtn` references, `just build-werror` path, and target-path mentions.

## Recommendation

**GO — technically feasible now, low risk, not urgent.** Nothing is blocking; the migration diff
is small and reversible. Given sbt 2 is ~1 month old, land it when convenient (or after the
scalafix delta is understood), rather than treating it as urgent. A "wait a quarter for the
ecosystem to settle" posture is also defensible but not required — every plugin this project uses
already ships sbt-2 artifacts.
